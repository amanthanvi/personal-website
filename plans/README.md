## Recommended order & dependency notes

Suggested sequence: **001 → 002 → 003 → 004** (default written non-interactively;
the top findings by leverage were turned into plans, per the `improve` skill's
"no user available to choose" fallback).

- **001 first** establishes a trustworthy signal: today the Astro type-check is
  `continue-on-error` in CI and absent from the deploy workflow, so nothing
  blocks a broken type from shipping. Landing it first means 002/003 get real
  CI verification. 001's own Step 1 aborts if the tree isn't currently green, so
  it can't turn `main` red by surprise.
- **002 and 003 are independent** of each other and of 004; either can go in any
  order after 001. 002 is the highest-volume cleanup (~1,000+ dead lines).
- **004 depends on 002 (recommended, not hard):** the doc corrections are
  accurate whether or not 002 has run, but reading cleanest if the orphaned
  boot/cursor files are already gone. If 002 hasn't landed, 004 says to note the
  still-present dead files in the PR.
- **Cross-cutting caveat:** 002 deletes `SandwormCursor` (dune) and
  `ArcadeEasterEggs` (arcade), which the design overhaul may also remove. 002 is
  written to no-op safely if a target file is already deleted, and to STOP if a
  target has been newly *mounted*.

## Direction (options for the maintainer, not ranked against the fixes above)

- **Blog is scaffolding, not a channel yet.** One post (`hello-world.mdx`), full
  MDX pipeline, reading-time, tags, blog route, and a "Blog" nav section all
  exist. `PRODUCT.md` names "research maturity" as a user need. The architecture
  is one or two posts away from being a real credibility surface (e.g. an
  IEEE CAI 2025 write-up — already cited in the hero). Grounding: existing but
  near-empty content collection + stated product intent. Trade-off: content
  effort, not engineering.
- **RSS + richer sitemap are one endpoint away.** The same static-endpoint
  pattern used in plan 003 would give the blog an RSS feed
  (`@astrojs/rss` is first-party). Low cost, only worth it once there are
  several posts. Grounding: blog collection already typed with `pubDate`.
- (Deliberately not suggested: "add analytics", "add dark mode", "add AI" —
  either already present, ungrounded, or contrary to `README`'s stated
  "no analytics/cookies" privacy stance.)

## Findings considered and rejected (so they aren't re-audited)

- **Font payload (finding #5): flagged, not planned — overlaps the design overhaul.** Marcellus/Silkscreen are theme fonts; whether they can be dropped depends on the light/dark theme collapse the sibling owns. Recommend the design branch trim the Google Fonts URL (`src/components/layout/BaseHead.astro`, the `fonts.googleapis.com/css2?...` link) to only the fonts the final theme set uses, and consider `media`/preload for the render-blocking `<link>`. Revisit as a plan only if the overhaul doesn't handle it.
- **Theme-system sprawl (dune/arcade CSS in `global.css`/`arcade.css`, `theme.ts`, `ThemeToggle.astro`, `DuneDunes.astro`):** owned by the parallel design overhaul; not planned here to avoid fighting it.
- **Zero automated tests:** for a static, no-input portfolio a unit/integration suite is low-value (YAGNI). The real verification need — a *blocking* check + build — is covered by plan 001. Not worth a separate test-suite plan.
- **Unused `--animate-*` tokens/keyframes** (`fade-in-up`, `glitch`, `pulse-glow`, `scan-line`, and the `circuit-draw`/`typing` keyframes) in `global.css`/`animations.css`: real dead CSS, but folded into plan 002's maintenance notes as a deferred follow-up because the design overhaul is actively rewriting `global.css` — pruning them now risks a merge collision.
- **`.hud-corners` / `.glow-text` unused utilities** (`hud.css`): documented HUD utilities (`DESIGN.md` §5) with no current users; left in place — low value to chase and the redesign may adopt them.
- **`astro:page-load` listeners without a `ClientRouter`/View Transitions** (in `index.astro`, `Header.astro`, `ThemeToggle.astro`, `ScrollProgress.astro`): the listeners never fire, but each script also initializes directly, so behavior is correct. Harmless belt-and-suspenders dead code; not worth a plan.
- **No theme switcher in the mobile nav** (`Header.astro` mobile bar omits `ThemeToggle`): plausibly by-design (`DESIGN.md` says the mobile bar is section-movement-focused) and squarely in the design overhaul's lane. Not planned.
- **`og-default.png` appears identical to `profile.jpg`** (same byte size) and **`site.webmanifest` `theme_color` (`#2ea3c4`) differs from `BaseHead` theme-color metas (`#f1f2f6`/`#080a12`)**: cosmetic; low impact. Not planned.
- **No CSP / security headers:** low value for a static site with no auth, forms, or user input, and GitHub Pages can't set response headers anyway. Considered and dropped.
- **Security scan:** no secrets found (the `contact@amanthanvi.com` address is intentionally public); all external links already use `rel="noopener noreferrer"`; no injection/user-input surface (static SSG). No prompt-injection content in the repo. No security findings.
